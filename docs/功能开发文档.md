# AgentFlow 功能开发文档

> 基于 LangGraph 构建的可视化 Agent 工作流编排平台（对标 Dify）
>
> 技术栈：Python、LangGraph、LangChain、MySQL、Redis、RAG、Tool Calling、StateGraph

---

## 0. 文档说明

本文档在初版「固定流程 Agent 系统」基础上做了**结构性升级**：AgentFlow 不再是一条写死的流水线，而是一个**用户可自行编排的可视化工作流平台**，核心能力对标 Dify。

新增/强化的核心需求：

1. **知识库上传 + 知识检索节点**：用户上传文档构建知识库，工作流中通过「知识检索节点」做 RAG。
2. **插件市场**：用户下载插件，每个插件即一个可拖入画布的「节点」。
3. **工作流市场**：用户下载现成工作流模板，一键导入。
4. **节点输入/输出字段规范**：节点间通过明确定义的 IO Schema 连线传值。
5. **自定义代码节点**：用户编写 Python 或 TS/JS 代码（添加前需选择语言），系统给出含输入输出的模板，用户只改核心逻辑。
6. **工作流调试日志**：测试运行时记录每个节点的输入、输出、状态、错误（仅测试态记录）。

> Dify 功能参考来源见文末「参考资料」。

## 1. 项目概述

### 1.1 项目定位

AgentFlow 是一个**可视化的 Agent 工作流编排平台**。用户在画布上拖拽节点、连线、配置 IO，即可编排出一条 AI 工作流；底层用 **LangGraph 的 StateGraph** 将这张可视化图编译为可执行的状态图运行。

平台定位三层：

| 层 | 角色 | 能力 |
|----|------|------|
| 编排层 | 普通用户 | 拖拽节点、连线、配置、调试、发布 |
| 运行层 | 运行引擎 | DSL → StateGraph 编译 → 状态驱动执行 |
| 生态层 | 开发者 | 发布插件（节点）、发布工作流模板到市场 |

### 1.2 与 Dify 的功能对标

## 3. 核心概念：工作流 DSL 与变量系统

平台的「灵魂」是一套 **Workflow DSL**——用户在画布上的所有操作最终序列化成这份 JSON，引擎据此编译成 LangGraph 的 StateGraph。

### 3.1 Workflow DSL 结构

```json
{
  "workflow_id": "wf_123",
  "name": "智能客服工作流",
  "version": 3,
  "graph": {
    "nodes": [
      {
        "id": "start_1",
        "type": "start",
        "data": {
          "outputs": [
            {"name": "query", "type": "string", "required": true}
          ]
        }
      },
      {
        "id": "kr_1",
        "type": "knowledge_retrieval",
        "data": {
          "dataset_ids": ["ds_001"],
          "top_k": 3,
          "score_threshold": 0.5,
          "inputs": [
            {"name": "query", "type": "string", "value": "{{start_1.query}}"}
          ],
          "outputs": [
            {"name": "result", "type": "array[object]"}
          ]
        }
      },
      {
        "id": "llm_1",
        "type": "llm",
        "data": {
          "model": "claude-opus-4-8",
          "prompt": "根据资料回答：{{kr_1.result}}\n问题：{{start_1.query}}",
          "outputs": [{"name": "text", "type": "string"}]
        }
      },
      {"id": "end_1", "type": "end",
       "data": {"inputs": [{"name": "answer", "value": "{{llm_1.text}}"}]}}
    ],
    "edges": [
      {"source": "start_1", "target": "kr_1"},
      {"source": "kr_1", "target": "llm_1"},
      {"source": "llm_1", "target": "end_1"}
    ]
  }
}
```

### 3.2 节点 IO 字段规范（重点需求）

每个节点都必须声明 `inputs` 和 `outputs` 的字段 schema，这是节点间连线传值的契约。

**字段定义：**

```python
from pydantic import BaseModel
from typing import Literal, Any

VarType = Literal["string", "number", "boolean", "object",
                  "array[string]", "array[object]", "file"]

class FieldSchema(BaseModel):
    name: str                      # 字段名
    type: VarType                  # 字段类型
    required: bool = False         # 是否必填
    description: str = ""          # 字段说明
    default: Any = None            # 默认值
    value: str | None = None       # 取值表达式，如 {{node_id.field}}

class NodeIOSpec(BaseModel):
    inputs: list[FieldSchema]      # 输入字段
    outputs: list[FieldSchema]     # 输出字段
```

**连线校验规则**（编辑期 / 编译期双校验）：

1. 下游节点 input 的 `value` 引用的 `{{node_id.field}}` 必须真实存在。
2. 引用字段的 **类型必须兼容**（如 string 不能连到 number 输入）。
3. 必填 input 必须有连线或默认值，否则编译报错。
4. 不允许引用「下游/未执行」节点的输出（DAG 拓扑校验）。

### 3.3 变量引用与解析

变量用 `{{node_id.field}}` 语法引用上游输出。运行时由变量解析器从 `WorkflowState.variables` 中取值并替换：

```python
import re

class VariableResolver:
    PATTERN = re.compile(r"\{\{(\w+)\.([\w\.]+)\}\}")

    def resolve(self, template: str, variables: dict) -> Any:
        """把 {{node.field}} 替换为变量池中的实际值"""
        def _sub(m):
            node_id, field = m.group(1), m.group(2)
            return str(variables.get(node_id, {}).get(field, ""))
        return self.PATTERN.sub(_sub, template)
```

### 3.4 WorkflowState（运行态变量池）

```python
from typing import TypedDict, Annotated
from operator import or_

class WorkflowState(TypedDict):
    workflow_id: str
    run_id: str
    mode: str                              # "debug" | "production"
    # 变量池：{node_id: {field: value}}，节点产出写入此处
    variables: Annotated[dict, or_]        # reducer 合并各节点输出
    # 控制
    current_node: str
    error: str | None
    # 调试日志（仅 debug 模式累积）
    node_logs: Annotated[list, lambda a, b: a + b]
```

> **设计要点**：变量池用 `Annotated[dict, or_]` 做 reducer，各节点只写自己的 `node_id` 命名空间，天然避免互相覆盖；这也是「节点 IO 隔离」的实现基础。

## 4. DSL → StateGraph 编译器

把可视化 DSL 编译成可执行的 LangGraph 图，是平台与「写死流程」最本质的区别。

```python
from langgraph.graph import StateGraph, START, END

class WorkflowCompiler:
    def __init__(self, registry: NodeRegistry):
        self.registry = registry

    def compile(self, dsl: dict):
        g = StateGraph(WorkflowState)
        graph = dsl["graph"]

        # 1. 注册节点：每个 DSL 节点 → 一个可执行函数
        for node in graph["nodes"]:
            node_impl = self.registry.create(node["type"], node["data"])
            g.add_node(node["id"], self._wrap(node_impl, node))

        # 2. 连边
        for node in graph["nodes"]:
            if node["type"] == "start":
                g.add_edge(START, node["id"])
            if node["type"] == "end":
                g.add_edge(node["id"], END)

        for edge in graph["edges"]:
            if edge.get("condition"):
                # 条件分支 → conditional_edges
                g.add_conditional_edges(edge["source"], self._cond(edge))
            else:
                g.add_edge(edge["source"], edge["target"])

        return g.compile(checkpointer=redis_checkpointer)

    def _wrap(self, node_impl, node):
        """包装节点：注入变量解析 + 调试日志埋点"""
        @trace_node(node["id"], node["type"])
        def runner(state: WorkflowState) -> dict:
            inputs = resolve_inputs(node["data"]["inputs"], state["variables"])
            outputs = node_impl.run(inputs, state)
            return {"variables": {node["id"]: outputs}}
        return runner
```

## 5. 内置节点库

对标 Dify 的节点体系。每个节点继承统一基类，声明 IO Schema。

### 5.1 节点基类与注册

```python
class BaseNode:
    type: str                     # 节点类型标识
    io_spec: NodeIOSpec           # 输入输出规范

    def run(self, inputs: dict, state: WorkflowState) -> dict:
        """执行节点逻辑，返回 outputs 字典"""
        raise NotImplementedError

class NodeRegistry:
    def __init__(self):
        self._types: dict[str, type[BaseNode]] = {}

    def register(self, node_cls: type[BaseNode]):
        self._types[node_cls.type] = node_cls

    def create(self, type_: str, data: dict) -> BaseNode:
        return self._types[type_](data)
```

### 5.2 内置节点清单

| 分类 | 节点 | 类型标识 | 说明 |
|------|------|---------|------|
| 起止 | 开始 | `start` | 定义工作流入参 |
| 起止 | 结束 | `end` | 定义工作流出参 |
| AI | LLM | `llm` | 调用大模型生成 |
| AI | 知识检索 | `knowledge_retrieval` | RAG 检索知识库 |
| AI | Agent | `agent` | 自主工具调用（增强）|
| 逻辑 | 条件分支 | `if_else` | IF/ELSE 多分支 |
| 逻辑 | 问题分类 | `classifier` | LLM 意图分类（增强）|
| 逻辑 | 迭代 | `iteration` | 遍历数组逐项处理 |
| 数据 | 代码执行 | `code` | 自定义 Py/TS 代码 |
| 数据 | 模板转换 | `template` | Jinja2 文本处理 |
| 数据 | 变量聚合 | `aggregator` | 多分支变量归一 |
| 数据 | 参数提取 | `param_extractor` | LLM 抽取结构化参数（增强）|
| 集成 | HTTP 请求 | `http_request` | 调外部 API |
| 集成 | 工具 | `tool` | 调用插件市场的工具插件 |

## 6. 知识库模块（上传 + RAG，重点需求）

对标 Dify 的知识库 RAG 管线：**Chunking → Indexing → Retrieval → Rerank → Generation**。

### 6.1 知识库构建流程

```
上传文档 → 解析 → 切分(Chunk) → 向量化(Embedding) → 建索引(Index) → 入库
   PDF/MD       提取文本   General/         embedding      高质量/经济     向量库+MySQL
   /DOCX/TXT              Parent-Child/QA   model
```

### 6.2 切分策略（对标 Dify 三种）

| 策略 | 适用 | 说明 |
|------|------|------|
| General（通用）| 普通文档 | 按分隔符 + 最大长度切分 |
| Parent-Child（父子）| 长技术文档 | 子块精检索，父块给上下文 |
| Q&A（问答）| 结构化表格/FAQ | 切成问答对 |

### 6.3 索引方法（对标 Dify 高质量/经济）

| 索引方法 | 实现 | 检索方式 | 适用 |
|----------|------|---------|------|
| 高质量（High-Quality）| Embedding 向量化入向量库 | 语义/混合检索 | 生产推荐 |
| 经济（Economy）| 关键词倒排索引 | 全文精确匹配 | 大批量低成本 |

> 注：高质量索引一经创建不可降级为经济（与 Dify 一致）。

### 6.4 检索设置（对标 Dify）

```python
class RetrievalConfig(BaseModel):
    search_mode: Literal["semantic", "keyword", "hybrid"] = "hybrid"
    top_k: int = 3                    # 召回数量
    score_threshold: float = 0.5      # 相似度阈值
    rerank_enabled: bool = False      # 是否启用 Rerank
    rerank_model: str | None = None
    # 混合检索权重
    semantic_weight: float = 0.7
    keyword_weight: float = 0.3
```

**混合检索（Hybrid Search）**：同时做向量检索 + 全文检索，再按权重融合或经 Rerank 重排，解决「纯向量检索匹配不到精确产品名/版本号」的问题。

### 6.5 知识库相关接口


## 7. 插件系统与插件市场（重点需求）

对标 Dify 插件生态：**插件 = 一个可拖入画布的节点**。用户从市场下载安装，安装后该节点出现在节点面板。

### 7.1 插件类型（对标 Dify 四类）

| 类型 | 说明 | 在画布中的体现 |
|------|------|---------------|
| Tool（工具）| 第三方服务/API 能力（天气、搜索、计算）| 工具节点 |
| Model（模型）| 模型提供商接入 | LLM 节点的模型选项 |
| Extension（扩展）| 简单 HTTP 服务扩展 | 自定义节点 |
| Agent Strategy（Agent 策略）| Agent 节点的推理决策逻辑（Function Calling/ReAct）| Agent 节点策略选项 |

### 7.2 插件规范（manifest）

每个插件是一个包，含描述文件 + 实现：

```yaml
# plugin.yaml
name: weather_tool
version: 1.0.0
type: tool                       # tool / model / extension / agent_strategy
author: someone
description: 查询城市天气
icon: weather.svg
entrypoint: main.py
# 节点 IO 规范——决定它在画布上长什么样、怎么连线
io_spec:
  inputs:
    - {name: city, type: string, required: true, description: 城市名}
  outputs:
    - {name: temperature, type: number}
    - {name: description, type: string}
```

```python
# main.py —— 插件实现
from agentflow.plugins import ToolPlugin, PluginResult

class WeatherTool(ToolPlugin):
    def invoke(self, inputs: dict) -> PluginResult:
        city = inputs["city"]
        data = call_weather_api(city)
        return PluginResult(outputs={
            "temperature": data["temp"],
            "description": data["desc"],
        })
```

## 8. 自定义代码节点 + 沙箱（重点需求）

用户在工作流中添加「代码节点」时：**先选语言（Python / TS-JS）→ 系统给出模板 → 用户只改核心逻辑**。

### 8.1 交互流程

```
点击「+ 代码节点」
   │
   ├─ 弹窗选择语言：[ Python ]  [ TypeScript / JavaScript ]
   │
   ▼
生成模板（含 main 函数签名 + 输入输出说明）
   │
   ▼
用户只编辑函数体核心逻辑
   │
   ▼
保存 → 运行时在沙箱中执行
```

### 8.2 代码模板

**Python 模板：**

```python
# 输入由上游节点变量映射而来，通过参数传入
# 输出必须是 dict，对应节点 outputs 定义的字段
def main(inputs: dict) -> dict:
    # ====== 在此编写你的核心逻辑 ======
    text = inputs["text"]
    result = text.upper()
    # =================================
    return {
        "output": result,   # 对应 outputs 字段
    }
```

**TypeScript / JavaScript 模板：**

```typescript
// inputs 来自上游节点；返回对象对应 outputs 字段
function main(inputs: Record<string, any>): Record<string, any> {
  // ====== 在此编写你的核心逻辑 ======
  const text = inputs["text"];
  const result = text.toUpperCase();
  // =================================
  return {
    output: result,
  };
}
```

### 8.3 IO 绑定

代码节点的 inputs/outputs 字段在节点配置面板中定义，运行时：
- 引擎按字段名把上游变量注入 `inputs`。
- 执行后取返回 dict 中对应 outputs 字段的值，写回变量池。
- 返回字段与 outputs 声明不符 → 记为节点错误。

### 8.4 沙箱执行（安全核心）

```python
class CodeNode(BaseNode):
    type = "code"

    def run(self, inputs, state):
        lang = self.data["language"]        # python / javascript
        code = self.data["code"]
        sandbox = PythonSandbox() if lang == "python" else NodeSandbox()
        try:
            outputs = sandbox.execute(
                code=code, entry="main", inputs=inputs,
                timeout=self.data.get("timeout", 10),
            )
        except SandboxTimeout:
            raise NodeError("代码执行超时")
        except SandboxError as e:
            raise NodeError(f"代码执行失败: {e}")
        return outputs
```

**沙箱安全约束：**

| 维度 | 限制 |
|------|------|
| 执行隔离 | 子进程 / 容器（如 gVisor、subprocess + 资源限制）|
| 超时 | 默认 10s，超时强杀 |
| 内存/CPU | cgroup 限额 |
| 网络 | 默认禁网（可白名单）|
| 文件 | 禁止访问宿主文件系统 |
| 导入 | Python 限制危险模块（os/subprocess/socket）|

> **可讲点**：代码节点是「用户提交任意代码」，安全是第一位的——必须沙箱隔离 + 资源限制 + 超时兜底，这是平台型产品的硬性要求。

用户把编排好的工作流发布为模板，他人可下载一键导入。本质是 **DSL 的导出 / 导入**。

### 9.1 导出 / 导入

```python
class WorkflowExporter:
    def export(self, workflow_id: str) -> dict:
        wf = repo.get_workflow(workflow_id)
        # 导出 DSL + 依赖声明（用到哪些插件、是否依赖某知识库）
        return {
            "dsl": wf.dsl,
            "dependencies": {
                "plugins": self._extract_plugins(wf.dsl),
                "datasets": self._extract_datasets(wf.dsl),  # 仅声明，不含数据
            },
            "meta": {"name": wf.name, "description": wf.description},
        }

class WorkflowImporter:
    def import_(self, template: dict, user_id: str) -> str:
        # 1. 检查依赖插件是否已安装，未装则提示安装
        self._check_dependencies(template["dependencies"], user_id)
        # 2. 重映射 ID，创建用户私有副本
```

- ID 重映射避免冲突。

## 10. 工作流调试日志（重点需求）

对标 Dify 的「Run History + 单步运行 + Last Run」。**仅在测试态（debug 模式）记录每个节点的输入、输出、状态、错误**；生产态默认不记录详细 IO（降本 + 隐私）。

### 10.1 记录内容

每个节点执行时记录一条 `NodeRunLog`：

```python
class NodeRunLog(BaseModel):
    run_id: str                  # 本次运行 ID
    workflow_id: str
    node_id: str
    node_type: str
    status: Literal["running", "succeeded", "failed"]
    inputs: dict                 # 节点输入（解析后的实际值）
    outputs: dict                # 节点输出
    error: str | None            # 错误信息（哪里出问题）
    error_node: bool             # 是否是错误源头节点
    elapsed_ms: int              # 耗时
    started_at: float
    finished_at: float
```

### 10.2 调试埋点（仅测试态）

```python
def trace_node(node_id: str, node_type: str):
    def deco(fn):
        @functools.wraps(fn)
        def wrapper(state: WorkflowState) -> dict:
            # 生产态跳过详细记录
            if state["mode"] != "debug":
                return fn(state)

            start = time.perf_counter()
            inputs_snap = copy.deepcopy(state["variables"])
            error = None
            try:
                out = fn(state)
                status = "succeeded"
            except NodeError as e:
                error, status, out = str(e), "failed", {"error": str(e)}
            elapsed = int((time.perf_counter() - start) * 1000)

            log = NodeRunLog(
                run_id=state["run_id"], workflow_id=state["workflow_id"],
                node_id=node_id, node_type=node_type, status=status,
                inputs=inputs_snap, outputs=out.get("variables", {}),
                error=error, error_node=bool(error), elapsed_ms=elapsed,
                started_at=start, finished_at=time.perf_counter(),
            )
            debug_logger.save(log)               # 落 MySQL
            out.setdefault("node_logs", []).append(log.dict())
            if error:
                raise NodeError(error)           # 继续上抛，触发流程级处理
            return out
        return wrapper
    return deco
```

### 10.3 单步运行（Single-Step Run）

对标 Dify：可单独测试某个节点，不跑整图。给定该节点的输入（手填或取自上次运行缓存），只执行这一个节点并返回结果，方便定位问题。

```
POST /api/v1/workflows/{id}/nodes/{node_id}/run
{ "inputs": { "query": "测试输入" } }
→ 返回该节点 outputs / status / error / elapsed_ms
```

### 10.4 运行历史与「Last Run」

- **Run History**：每次测试运行持久化完整执行轨迹，可回放查看「数据如何在节点间流动」。
- **Last Run**：每个节点缓存最近一次成功执行的 IO，画布上点节点即可查看，无需重跑（节省 LLM 调用成本）。

### 10.5 接口与数据表

```

GET  /api/v1/runs/{run_id}/nodes/{nid}   # 单节点 Last Run
```
